# 村域图斑分区两种方案对比：深度学习笔记

---

## 第一部分：问题的本质

### 一句话描述
> 给定一个大多边形（村边界）和其内部的N个小多边形（图斑），将大多边形切分为若干子区域，每个子区域包含≤K个图斑，且切分不能破坏任何图斑。

### 硬约束 vs 软目标

**硬约束**（必须满足）：
- 每个子区域图斑数 ≤ 阈值K
- 切割不穿过任何图斑（拓扑安全）
- 子区域之间不重叠
- 子区域的并集 = 原村边界

**软目标**（尽量优化）：
- 各子区域图斑数尽量均匀
- 子区域形状尽量紧凑
- 切割线尽量短/简单

### 从一维到二维的思维跳跃

```
一维：线段上的分组
  N个小线段 → 排序 → 在间隙处切一刀 → 完成
  关键约束：只能在间隙处切

二维：多边形中的分区（本题）
  N个小多边形 → ??? → 在"间隙"处切割 → 完成
  关键约束：同上，但"间隙"变成了二维的缝隙
```

二维的"间隙"怎么找？有两种完全不同的思路：

| 思路 | 方案 | 怎么找"间隙" |
|------|------|-------------|
| **在几何空间找** | 坐标递归二分 | 投影到轴上→合并区间→区间间的间隙 |
| **在图空间找** | 图划分 | 邻接图上的切边=相邻图斑间的边界 |

## 第二部分：方案A — 坐标递归二分（拓扑安全版）

### 核心流程图

```
输入: 区域R, 图斑集P, 阈值K
      │
      ▼
  |P| ≤ K? ──是──► 输出叶子节点，结束
      │否
      ▼
  选择切割轴（包围盒较长方向）
      │
      ▼
  将每个图斑投影到轴上 → 得到N个区间[min, max]
      │
      ▼
  ┌─────────────────────────────────┐
  │  排序区间 → 合并重叠 → 找间隙    │ ← 拓扑安全的核心！
  │                                   │
  │  ·─·  ·──·   ·─·  ·──·           │
  │  └合并┘  间隙  └合并┘              │
  │            ↑                       │
  │         安全切割位置               │
  └─────────────────────────────────┘
      │
      ▼
  选择最接近中位数的间隙中心作为切割点
      │
      ▼
  用半平面求交切割区域多边形
      │
      ├── 左半部分 → 递归处理
      └── 右半部分 → 递归处理
```

### 拓扑安全的关键：投影区间间隙法

这是整个算法的核心创新。用一个例子说明：

```
想象从上往下看，5个图斑在X轴上的投影：

X轴: 0     10    20    30    40    50    60    70    80

图斑1: [2━━━━━12]
图斑2:     [8━━━━━━━25]       ← 注意和图斑1有重叠
图斑3:                 [30━━━━━━48]
图斑4:                     [35━━━━━━55]  ← 和图斑3有重叠  
图斑5:                                    [62━━━━75]

合并后:  [2━━━━━━━━━25]      [30━━━━━━━━━55]    [62━━75]
                       ↑                    ↑
                  间隙[25,30]          间隙[55,62]
                  中心=27.5            中心=58.5

如果中位数质心在35附近 → 选间隙[25,30]，切在27.5
```

为什么这样做一定拓扑安全？因为间隙意味着在这个X坐标范围内，**没有任何图斑存在**，所以垂直于X轴的切割线不会穿过任何图斑。

### 两个轴都找不到间隙怎么办？

理论上可能发生：当图斑在两个方向上都完全覆盖了区域。但在实际地块数据中，道路、水渠、空地等总会留下间隙。如果真的找不到，算法退化为不切分（保留为一个区域），这是保守但安全的策略。

### 优势与局限

**优势**：
- 实现清晰，逻辑链路短
- 几何输出是直接的（切割线 → 两个子多边形）
- O(N log N) 时间复杂度
- 切割线是直线，视觉上规整

**局限**：
- 切割线方向固定（只能水平/垂直），不够灵活
- 不考虑图斑之间的邻接关系
- 可能将两个紧密相连的图斑分到不同区域
- 对L形或C形区域效果差

## 第三部分：方案B — 多层级图划分（METIS思想）

### 核心洞察

**将"在哪里切"的问题转化为"哪些图斑分到一组"的问题。**

一旦做了这个转化，拓扑安全就是自动保证的——我们不需要任何切割线，只需要给每个图斑一个分组编号。

### 三阶段流程

```
原始图(N=10000节点)
      │
      ▼ Phase 1: 粗化
合并相邻节点对 ──► 5000节点 ──► 2500 ──► ... ──► 50节点
      │                    使用Heavy Edge Matching
      ▼                    (合并共边最长的相邻对)
      │
      ▼ Phase 2: 初始划分
在50节点的小图上 ──► 坐标排序二分 ──► 初始分组
      │
      ▼ Phase 3: 反粗化 + 细化
50节点 ──展开──► 100节点 ──KL细化──► ... ──► 10000节点
              每层都做KL/FM优化
              交换边界节点减少切边
```

### 为什么要粗化？—— 多层级的精妙之处

直觉解释：

```
原始图：10000个节点，做KL优化需要检查大量节点对 → 很慢
粗图：50个节点，做KL优化几乎瞬间完成

但粗图的划分质量够好吗？
→ 够好！因为粗化过程中合并了紧密连接的节点，保留了全局结构
→ 不够精确？没关系，展开时用KL逐层修正局部细节

这就是多层级方法的核心智慧：
  全局结构在粗图上决定（快速），
  局部细节在细图上修正（精确）。
```

### Heavy Edge Matching (HEM)

```
粗化的一步：

  A ─3─ B ─5─ C ─2─ D
  │     │     │     │
  1     4     1     3
  │     │     │     │
  E ─2─ F ─6─ G ─1─ H
  
  边上数字=共边长度（边权）

HEM过程：
  1. 随机选未匹配节点F
  2. F的邻居中最重边是G(权6) → 匹配 F+G → 合并为FG
  3. 选未匹配节点B
  4. B的邻居中最重边是... C(权5) → 匹配 B+C → 合并为BC
  5. ...继续

结果：
  A ─── BC ─── D        节点数减半
  │     │      │        但保留了重要连接
  E ─── FG ─── H

为什么选最重边：
  重边 = 共边长 = 紧密相连 = 很可能在同一分组
  先合并它们，就不需要在划分时再考虑它们
```

### KL细化

```
划分后的边界：                   KL交换后：

  A₁ A₁ A₁ │ B₁ B₁              A₁ A₁ A₁ │ B₁ B₁
  A₁ A₁ B₁ │ B₁ B₁     →        A₁ A₁ A₁ │ B₁ B₁
  A₁ A₁ A₁ │ B₁ B₁              A₁ A₁ A₁ │ B₁ B₁
             ↑                               ↑
  注意这个B₁被错误地                 交换回A₁后，切边减少了
  分到了A的区域中

增益计算：
  对节点u在组A，考虑移到组B：
    external(u) = u与B组邻居的连边数 = 移动后减少的切边
    internal(u) = u与A组邻居的连边数 = 移动后增加的切边
    gain(u) = external(u) - internal(u)
    
  gain > 0 → 移动有收益 → 执行
```

### 几何恢复：从分组到多边形

图划分给出的是分组编号，但我们需要的是子区域多边形。这是图论方法的主要复杂性所在：

```
分组结果：
  ┌─A─A─A─┐
  │ A A A  │
  │ A B B  │ ← 分组编号
  │ B B B  │
  └─B─B─B─┘

需要恢复为：
  ┌─────────┐
  │ Region A│
  │─────────│ ← 这条边界线怎么确定？
  │ Region B│
  └─────────┘

策略：
  1. 每组图斑 → Union → 得到组的"图斑区域"
  2. 图斑间的间隙 → 按最近距离分配
  3. 各组"图斑区域" + 间隙 → 与村边界求交 → 最终子区域
```

### 优势与局限

**优势**：
- 天然拓扑安全（不存在切割线概念）
- 切边最小化 → 区域更自然
- 多层级策略处理大规模数据高效
- 区域形状贴合图斑实际分布

**局限**：
- 几何恢复复杂，可能产生不规则边界
- 实现复杂度高
- 需要建立邻接图（O(N log N)但常数较大）
- 参数敏感（粗化策略、细化轮数等）

## 第四部分：判断最优的量化指标

```
Score = w₁ × 均衡度⁻¹ + w₂ × 紧凑度 + w₃ × 切边代价⁻¹ + w₄ × 面积均衡度⁻¹

其中：
  均衡度 = max(图斑数) / avg(图斑数)     →  1.0 = 完美
  紧凑度 = avg(4πA/P²)                  →  1.0 = 圆形
  切边代价 = 被切开的相邻图斑对数         →  0 = 完美(不可能)
  面积均衡度 = max(面积) / avg(面积)      →  1.0 = 完美
```

实际项目中如何选择权重：
- **行政管理场景**：均衡度权重最高（每个网格员管理的地块数要均匀）
- **物流配送场景**：面积均衡度重要（每个配送区域的物理面积要均匀）
- **自然资源管理**：紧凑度重要（区域形状要自然，不要细长碎片）

## 第五部分：核心知识点详解

### 5.1 STR-tree 的工作原理

```
普通R-tree（逐个插入）：
  包围盒重叠严重 → 查询时需要遍历多个分支

STR-tree（批量加载）：
  Step 1: 按X坐标排序所有几何体
  Step 2: 每 ceil(√N) 个一组（Tile）
  Step 3: 每组内按Y坐标排序
  Step 4: 再分组，递归构建

  结果：包围盒重叠极少 → 查询效率高

为什么我们用STR-tree而不是普通R-tree：
  我们的场景是"所有图斑已知 → 大量邻接查询"
  = "先全部插入 → 后批量查询"
  = STR-tree的最佳使用场景
```

### 5.2 半平面求交 vs 直接Split

```
为什么JTS没有polygon.split(line)?

因为一条线可能把多边形切成2个、3个甚至更多部分（凹多边形的情况）。
split操作的结果不确定，API设计困难。

半平面求交的思路：
  原多边形 ∩ 左半平面 = 左半部分
  原多边形 ∩ 右半平面 = 右半部分

  半平面 = 一个非常大的矩形，覆盖切割线的一侧

  polygon.intersection(bigRectangle) 
  → 这是JTS最核心、最稳健的操作之一
  → 经过20年的bug修复和边界情况处理
  → 比任何自定义split实现都可靠
```

### 5.3 Kernighan-Lin算法的直觉

```
想象你有一堆围棋棋子在棋盘上，分成黑白两组。
"切边" = 黑子和白子相邻的地方。

KL算法就像是：
  反复地找一个黑子和一个白子交换位置，
  使得交换后黑白边界更短。

  为什么有效：
  每次交换都贪心地选最大改善，
  但允许暂时变差（避免局部最优），
  最后选整个过程中最好的状态。

  这就是KL的精妙之处：
  它不是纯贪心，而是带"回溯"的贪心。
```

## 第七部分：思维内化

### 解题思维模型

```
遇到空间约束优化问题时：

1. 降维思考
   二维太复杂？先想一维情形
   核心约束在一维中是什么样的？

2. 域转换
   几何域 → 图论域（或反向）
   问自己：哪个域更容易处理这个约束？

3. 分治递归
   大问题无从下手？每次只做一个二分决策
   log层就够了

4. 先松弛后修复
   先忽略难约束，得到初步解
   再修复违反约束的部分
   （投影间隙法就是这个思路的极致体现）

5. 多层级
   大规模直接优化太慢？
   粗化→在小图上决策→展开时修正
```

这些思维模型不只适用于本题，它们是算法设计中最通用的武器。
